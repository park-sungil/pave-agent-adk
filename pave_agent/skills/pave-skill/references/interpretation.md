# Interpret 응답 포맷 규칙

> 이 문서는 **interpret 응답의 형식**만 규정한다. 도메인 지식(파라미터 의미, 트레이드오프, PVT 상관관계 등)은 `pave_domain.md`에 있으며, interpret 호출 시 관련 섹션만 자동 주입된다.

## 기본 원칙

- **질문에만 답한다**. 사용자가 묻지 않은 metric, trade-off, 권장사항을 추가하지 않는다.
- **데이터의 값만 사용한다**. 계산, 추정, 보간 금지.
- **한국어로 답변한다**.
- 응답은 간결하게. 한 줄로 끝나는 질문에는 한 줄로 답한다.

## 질문 유형별 포맷

### 1. 특정 metric 조회
예: "주파수가 얼마야?", "S_POWER 알려줘"

- 질문한 **그 metric만** 표로 제시
- 단위 포함 (GHz, mW, nA 등)
- 필요한 경우 1~2줄 간단 설명 (조건, PDK 정보 등)
- trade-off, 다른 metric 언급, 설계 권장사항 **금지**

### 2. 여러 metric 조회
예: "주파수와 power 알려줘"

- 사용자가 명시한 metric만 표로 제시
- 필요시 metric 간 관계 한 줄 언급 가능

### 3. 벤치마킹 비교 (두 PDK 간 비교)
예: "A PDK vs B PDK", "3nm → 2nm 변화"

**Summary**
- 핵심 변화를 1~3줄로 요약
- 사용자가 명시한 metric 중심 (불필요한 metric 언급 금지)

**Table**
- Metric, A 값, B 값, Delta, Delta(%)
- analyze가 제공한 숫자 그대로 사용

**Technical Insight** (사용자가 해석을 원할 때만)
- BSIM-CMG 등 도메인 지식으로 변화의 원인 설명
- trade-off 언급은 사용자가 물어본 metric 내에서만

### 4. 전체 요약 요청
예: "전체 보여줘", "전반적으로 어때"

- 모든 주요 metric을 테이블로 정리
- 각 metric별 핵심 수치만, 긴 해석 없이

### 5. 도메인 개념 질문
예: "Temperature Inversion이 뭐야?", "SSPG가 뭐야?"

- pave_domain.md의 해당 섹션 내용을 바탕으로 설명
- 사용자의 현재 데이터와 연관 지을 수 있으면 간단히 연결

## 금지 사항

- **숫자 계산 금지**: delta, %, 평균 등 어떤 산술도 interpret가 직접 하지 않는다. analyze가 제공한 값을 그대로 사용한다.
- **데이터에 없는 metric 추가 금지**: 데이터에 없는 값을 추정, 보간, 임의 생성하지 않는다.
- **사용자가 묻지 않은 내용 추가 금지**: trade-off, 설계 권장사항, "참고로…" 등의 사족 금지.
- **raw 에러/필드명 노출 금지**: "error", "needs_input", JSON 필드명 등을 사용자 응답에 노출하지 않는다.
